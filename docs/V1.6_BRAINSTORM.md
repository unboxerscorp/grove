# grove v1.6+ 브레인스토밍

> Status: historical v1 brainstorm; superseded by the current v2 live cockpit model documented in `docs/design/MASTER_NODE.md` and `docs/agents/LEAD-HANDOFF.md`.
> 작성일: 2026-06-04

## 전제

- v1.0~v1.4는 제품 골격, 하드닝, 신뢰성, 멀티 오케스트레이터, 팀 surface를 단계적으로 출시했다.
- v1.5는 mock drift를 막는 실제 서버 e2e, dashboard delegate, node-status와 board replay polish를 진행 중이다.
- v1.6+는 "보이는 실제 CLI 세션으로 움직이는 24/7 팀-OS"를 더 오래, 더 여러 사람이, 더 적은 수동 조작으로 운영하게 만드는 단계다.
- 장기 운영 모델은 로컬-퍼-멤버 실행, Tailscale을 통한 선택적 공유, 보드=위임 프로토콜, 조직도와 감사 로그를 중심으로 한 작업 가시성이다.
- v1.6은 새 대형 플랫폼을 벌리기보다 v1.5까지 생긴 surfaces를 묶어 "하루 종일 켜 두고 일할 수 있는 팀룸"을 만드는 데 초점을 둔다.

## 우선순위 기준

- **P0**: v1.6 후보. 24/7 운영, 멀티 오케스트레이터, 팀 협업을 바로 개선한다.
- **P1**: v1.6 stretch 또는 v1.7 후보. 가치가 크지만 의존성이 더 많다.
- **P2**: v1.7+ backlog. 제품성을 넓히거나 고도화가 필요하다.
- **P3**: 장기 탐색. 방향성은 좋지만 아직 요구가 선명하지 않다.

## 제안 v1.6 스코프

v1.6의 이름은 **"운영 가능한 팀룸"**이 적합하다. v1.5가 계약 drift를 막고 dashboard delegate를 닫는다면, v1.6은 하루 종일 켜 둔 room에서 사람이 놓치지 않고, node가 죽어도 복구되고, 여러 멤버가 동시에 봐도 상태가 정리되는 수준을 목표로 한다.

### v1.6 핵심 항목

1. **위임 체인 탐색기 + 서브리드 워크로드 패널**
   - audit/board/org를 묶어 delegation lineage, subtree workload, blocked aging을 보여준다.
   - multi-orchestrator가 "가능"한 상태에서 "운영 가능한" 상태로 넘어가는 핵심 surface다.

2. **presence + 알림 규칙 엔진**
   - 누가 board/task/org를 보는지 표시하고, blocked/dead/stale/cost unknown/mention을 dashboard와 Slack으로 알린다.
   - v1.6의 협업 MVP이며 ask-human과 team auth를 실제 사용 흐름으로 연결한다.

3. **node auto-recovery policy + room supervisor**
   - pane missing, transcript stale, connector down을 감지해 repair/restart 후보를 만들고, web/executor/connector/health watcher를 한 lifecycle로 관리한다.
   - 24/7 운영의 신뢰성 핵심이다.

4. **project import/export MVP + 온보딩 위저드 v2**
   - 새 멤버가 template에서 room을 만들고 smoke task까지 실행하며, 기존 room을 portable bundle로 옮길 수 있게 한다.
   - 로컬-퍼-멤버 모델의 재사용성을 만든다.

5. **self-retro lane v0**
   - 하루 또는 release 단위로 audit/blocked/cost unknown을 요약하고 개선 task를 만든다.
   - v1.6에서 AI 활용을 작게 증명하고, v1.7의 자율 pickup과 routing recommender로 연결한다.

### v1.6 exit criteria

1. lead가 dashboard에서 lead -> sublead -> maker 위임 체인과 각 subtree의 ready/running/blocked 상태를 한 화면에서 확인한다.
2. 두 명 이상의 member가 같은 room에 접속했을 때 presence가 표시되고, task 편집/상태 변경 actor가 audit에 남는다.
3. blocked aging, dead pane, stale claim, cost unknown 중 최소 4개 알림이 dashboard에 뜨고 Slack connector가 켜진 경우 thread/channel로도 전달된다.
4. tmux pane kill 또는 transcript stale을 test scenario로 넣었을 때 node auto-recovery가 detect, suggest/repair, audit record까지 수행한다.
5. `room supervisor`가 web, pull executor, channel connector, health watcher를 시작/중지하고 health를 dashboard에 노출한다.
6. onboarding wizard가 template 선택, CLI/auth/tmux check, first smoke task, dashboard open까지 안내한다.
7. project export/import가 org + board snapshot + role/template metadata를 redacted bundle로 round-trip한다.
8. self-retro lane이 audit/board를 읽어 최소 1개의 actionable improvement task 또는 comment를 생성한다.
9. full check와 실제 서버 e2e가 green이고, 새 surfaces는 mock-only contract에 의존하지 않는다.

## 리스크와 판단

- multi-machine aggregation, cross-machine executor, command-heavy Slack은 가치가 크지만 보안·동기화·권한 모델이 커진다. v1.6은 로컬 room의 24/7 품질을 먼저 닫고 v1.7로 넘긴다.
- autonomous pickup은 장기 핵심이지만, v1.6에서는 self-retro와 work-pickup 제안 수준이 안전하다. 실제 자동 claim 확대는 backpressure와 recovery가 안정된 뒤가 맞다.
- template marketplace는 import/export가 먼저 있어야 한다. v1.6은 portable bundle을 만들고 marketplace는 v1.7에 얹는다.
- 모바일 view는 제품성이 높지만 v1.6의 핵심 병목은 dead/blocked를 놓치는 운영 리스크다. 알림 규칙과 presence가 먼저다.
